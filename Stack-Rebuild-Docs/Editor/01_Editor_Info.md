# Editor Architecture and Capabilities

The `Editor` module is the original non-destructive, node-based image-editing engine of Modular Studio. In the Stack rebuild, this system must behave exactly as the web counterpart, executing an OpenGL rendering loop based on a strictly sequential, registry-driven layer stack. 

**Core Rendering Philosophy:** The engine is fundamentally designed around linear sequences. Tools are not globally active overlays; they are discrete steps applied in order. Layer 2 can ONLY see the output generated by Layer 1. Layer 3 can ONLY see the output of Layer 2, and so on. This strict chronological processing dictates how the Editor tab must be laid out and how users must interact with their workflows.

## 1. The Persistent Base Canvas
Unlike conventional photo editors where the image dictates the document size, the Editor maintains a persistent **Base Canvas** (`state.document.base`).
- **Source Independence:** Users can build text-only or generated noise projects without importing any source image.
- **Image Fitting:** When an image is loaded, users choose between updating the canvas to the image's dimensions or bringing the image into the pre-existing canvas bounds.
- **Resolution Control:** The UI exposes *Resolution Scale*, *Expander*, and *Crop / Transform* layers to destructively modify the canvas bounds downstream within the layer stack, rather than changing the source image size directly.

## 2. Editor Layout & Navigation
The UI uses a neumorphic left-docked sidebar and an aspect-fitted, rounded inset viewport. The sidebar navigation consists of five main tabs:

1. **Layers (Category Hub):** 
   - Displays all available tools categorized into `Base`, `Color`, `Texture`, `Optics`, `Stylize`, and `Damage` groups.
   - Presents a grid of "Effect Chips". Actions allow adding layers or selecting active layer families.
2. **Canvas (Global State):**
   - Owns base canvas details, resolution dimensions, zoom controls, export behaviors (8-bit vs. 16-bit PNG), and background mode (Transparent or Solid Color).
   - Manages replacing or removing the current source image. 
3. **Selected (Contextual Inspector):**
   - Renders the custom properties and sliders for the actively selected layer in the stack. 
   - Supports diverse control primitives: Range sliders, Checkboxes, 3-Way Color Wheels (`colorWheel3Way`), Palettes (`paletteEditor`), and specialized editors (DNG Develop, Text Overlay, Background Patcher).
4. **Pipeline (Layer Stack Management):**
   - Displays the active layers as a linear, chronological list representing the exact rendering history. Because execution is sequential (Layer N+1 only sees Layer N), this panel is critical.
   - Supports drag-and-drop reordering (which immediately restructures the rendering sequence), visibility toggling, duplication, and layer removal.
5. **Scopes (Image Diagnostics):**
   - Displays histograms, vectorscopes, RGB parades, and brightness/saturation readouts. 
   - Provides Split / Hover Compare options against the original unprocessed image.

## 3. Specialized Layer Editors

### A. DNG Develop (Low Priority / Future Scope)
*Note: Ensure the architecture has placeholder room to inject a DNG container at the absolute bottom of the node stack, but implementing actual DNG decoding and processing is explicitly NOT a priority during the initial C++ build. It should be skipped until core features are completely stable and until the user says to work on it.*
- **Expected Integration:** Automatically injected as the very first chronological node in the layer stack when a `.dng` (RAW) image is loaded.
- **Worker/GPU Architecture:** The RAW container parsing and initial mosaic decode are performed asynchronously via background threads. The decoded buffer is then piped into the GPU pipeline so adjustments occur live interactively.
- **Controls & Tabs:** Will expose multiple sections including Interpretation, Normalize, Remosaic/Demosaic quality, Color/WB, Tone, Noise/Detail, and embedded Corrections.
- **Feedback:** Displays an embedded "Rendering Badge" on the canvas during heavy rebuilds and handles 16-bit PNG exports seamlessly.

### B. Text Overlays (`textOverlay`)
- **Render Path:** Text is rasterized to a surface first, then composited natively into the WebGL node chain. This allows downstream modules like color grades or crop adjustments to perfectly alter the text.
- **Interactive UX:** Moving, rotating, and resizing text does not rely on a side-panel. Instead, manipulating the text box natively acts via inverse transforms aligned correctly to the engine's downscaled-for-performance working resolution.
- **Params:** Includes inputs for Text Content, Font Family, Font Size, Color, Opacity, Rotation, X, and Y positions.

### C. Background Patcher (`bgPatcher`)
- **Purpose:** Specifically built to generate perfect transparency mats and patch artifacts in background-removals.
- **Tools:** Includes eyedropper flood picking, color tolerance spectrums, contiguous filling, and edge-smoothing (Defringe, Spread Radius).
- **Manual Adjustments:** Offers an active brush mode for painting transparency or masking (Keep vs Remove) and "Patching Mode" drawing squares over remaining artifact blobs.
- **Tolerance Spectrum:** An in-UI visual strip displaying the active range of smoothed colors being keyed out.

### D. Expander module
- **Purpose:** Natively grows the preview and export resolution bounds non-destructively.
- **Capabilities:** Users control uniform padding distances (up to 4096px bounds) and choose RGBA fill color (or transparency) via custom controls.

## 4. Viewport Interactions
- **Aspect-Fitted Stage:** Ensures images do not arbitrarily stretch. It perfectly fits within a rounded container.
- **Gizmos & Overlays:** The viewport responds directly to specific active layers displaying:
  - Text overlay rotate/resize bounding box handles.
  - Tilt-shift or Chromatic Aberration positional focal pins.
  - Background Patcher selection boxes and active Loupe display.
- **Compare:** A "Hover Compare" layer is instantly accessible, falling back to the raw source (`hoverSource`) when enabled.

## 5. Storage / Document Payload 
The Editor handles a persistent JSON state payload composed of:
- `base`: Contains width, height, backgroundMode, backgroundColor.
- `source`: Appended only if a source image exists. Embedded `source.imageData` or `source.dng.attachmentId`.
- `layerStack`: The active rendering instructions.
- `preview`: Automatically commits a low-res base64 representation of the work on save to assist Library grid view caching.
